# 原子工艺模板 - 需求规格说明书

## 1. 项目概述

### 1.1 背景与目标

re-flow 已实现 RSS 订阅管理和 AI 配置管理功能。本功能旨在将两者结合，构建一个**原子工艺模板系统**，让用户能够：

- 创建可复用的 AI 处理模板（工艺模板）
- 将多个模板组合成处理管道（Pipeline）
- 对 RSS 文章应用模板或管道进行处理
- 灵活切换查看处理结果和原文

**灵感来源**：参考 [FeedCraft](https://github.com/Colin-XKL/FeedCraft) 的设计理念
- CraftAtom（工艺）：单个处理操作
- CraftFlow（工艺组合）：多个工艺组成的序列
- Recipe（食谱）：指定处理规则

### 1.2 核心概念

| 概念 | 说明 | 示例 |
|------|------|------|
| **工艺模板 (Craft Template)** | 单个 AI 处理单元，包含 Prompt 模板和关联的 AI 配置 | 翻译为中文、生成摘要、提取关键词 |
| **管道 (Pipeline)** | 多个工艺模板按顺序组合的处理流程 | 全文提取 → 摘要 → 翻译 |
| **处理结果 (Processing Result)** | 对文章应用模板/管道后产生的输出 | 处理后的文本内容 |

### 1.3 技术选型

- **AI SDK**: Vercel AI SDK（复用现有 AI 配置系统）
- **数据库**: Drizzle ORM + SQLite
- **前端框架**: Next.js 16 (App Router) + React 19
- **样式**: Tailwind CSS 4
- **认证**: JWT + HTTP-only cookie（复用现有机制）
- **拖拽**: @dnd-kit/core + @dnd-kit/sortable

### 1.4 设计决策

| 决策项 | 决策内容 | 说明 |
|--------|----------|------|
| Content 变量 | 只用 `{{content}}`，直接传递原始 HTML | 现代 LLM 可直接理解 HTML，无需额外处理 |
| Pipeline 错误处理 | 整体失败 | 任一步骤失败则标记整个处理为 error |
| 存储清理 | 暂不处理 | 全量存储处理历史，后续再考虑清理策略 |
| 批量并发控制 | 服务端 API 内控制 | 在 POST /api/process 中限制并发数为 3，无需独立 Worker |
| Streaming | 不需要 | 一次性输出完整结果 |
| AI Config 删除保护 | API 预检查 | 删除前检查关联模板，返回友好提示和关联列表 |
| Pipeline 编辑器 | 拖拽排序 | 使用 @dnd-kit 实现可视化步骤排序 |
| 预设模板 | 代码定义 + 复制使用 | 预设模板在代码中定义，UI 展示"模板库"，用户使用时复制到自己的模板 |
| 处理入口 | 文章详情页优先 | 先实现单篇处理，批量处理在 Phase 2 |

### 2.2 功能详细描述

#### 2.2.1 工艺模板管理

**模板列表**：
- 展示当前用户的所有工艺模板
- 卡片式或列表式布局
- 每个模板显示：
  - 模板名称
  - 分类标签（摘要、翻译、过滤等）
  - 关联的 AI 配置名称
  - Prompt 模板预览
- 支持编辑和删除操作

**创建/编辑模板**：
- 模板名称（必填，3-50 字符）
- 模板描述（可选）
- 分类选择（下拉：摘要/翻译/过滤/分析/改写/自定义）
- 关联 AI 配置（下拉选择已配置的 AI）
- Prompt 模板（必填，多行文本编辑器）
  - 提供变量提示和自动补全
  - 支持预览渲染效果

**Prompt 模板变量系统**：

| 变量 | 来源 | 说明 |
|------|------|------|
| `{{title}}` | RSS Item | 文章标题 |
| `{{content}}` | RSS Item | 文章内容（原始 HTML，AI 可直接理解） |
| `{{author}}` | RSS Item | 作者 |
| `{{link}}` | RSS Item | 原文链接 |
| `{{publishedAt}}` | RSS Item | 发布时间 |
| `{{readingTime}}` | RSS Item | 预计阅读时长（分钟） |
| `{{feedTitle}}` | Feed | 订阅源名称 |
| `{{feedUrl}}` | Feed | 订阅源地址 |
| `{{prev_output}}` | Pipeline | 上一步的输出（仅管道中的模板可用） |

**预设模板**（代码定义，UI 展示"模板库"）：

用户可以从预设模板库中选择并复制到自己的模板列表，预设模板始终保持干净不可修改。

| 模板名称 | 分类 | Prompt 模板 |
|---------|------|-------------|
| 文章摘要 | summarize | `请为以下文章生成一个简洁的摘要（100字以内）：\n\n标题：{{title}}\n内容：{{content}}\n\n摘要：` |
| 翻译为中文 | translate | `将以下内容翻译成简体中文，保持原文的语气和风格：\n\n{{content}}` |
| 翻译标题 | translate | `将以下标题翻译成简体中文：\n\n{{title}}` |
| 关键词提取 | analyze | `从以下文章中提取5个关键词，以JSON数组格式返回：\n\n{{content}}\n\n关键词：` |
| 过滤软文 | filter | `判断以下文章是否值得阅读。如果是有价值的科技资讯，回复"KEEP"；如果是营销软文，回复"SKIP"。\n\n标题：{{title}}\n内容：{{content}}\n\n判断结果：` |
| 重写标题 | rewrite | `请为以下文章生成一个更吸引人的标题（不超过30字）：\n\n原标题：{{title}}\n内容摘要：{{content}}\n\n新标题：` |

#### 2.2.2 管道管理

**管道列表**：
- 展示当前用户的所有管道
- 显示管道名称、描述、包含的步骤数量
- 支持编辑和删除操作

**创建/编辑管道**：
- 管道名称（必填，3-50 字符）
- 管道描述（可选）
- 步骤编辑器：
  - 从模板列表中选择添加步骤
  - **拖拽排序**步骤顺序（使用 @dnd-kit）
  - 可视化展示处理流程
  - 删除单个步骤

**管道可视化**：
```
┌──────────┐    ┌──────────┐    ┌──────────┐
│  摘要    │ →  │  翻译    │ →  │  格式化  │
└──────────┘    └──────────┘    └──────────┘
    ↓               ↓               ↓
 输出1           输出2           最终输出
```

#### 2.2.3 文章处理

**单篇处理**：
- 在文章详情页或列表页提供"处理"按钮
- 弹窗选择：使用单个模板 或 使用管道
- 显示处理进度
- 处理完成后自动切换到结果视图

**批量处理**：
- 在文章列表支持多选
- 批量处理 = 并发触发多个独立的单篇处理
- 每篇文章独立调用 AI，互不干扰
- **服务端 API 控制并发数（限制为 3）**，避免 API 过载
- 显示处理进度：已完成/总数、成功/失败

**处理流程**：
```
用户选择文章 + 选择模板/管道
         ↓
    创建 ProcessingResult (status: pending)
         ↓
    执行处理：
    ┌─────────────────────────────────────┐
    │ Step 1: 获取 RSS Item 内容          │
    │         ↓                           │
    │ Step 2: 渲染 Prompt 模板            │
    │         {{title}} → 实际标题        │
    │         {{content}} → 实际内容      │
    │         ↓                           │
    │ Step 3: 调用 AI API                 │
    │         使用关联的 AI Config        │
    │         ↓                           │
    │ Step 4: 保存输出                    │
    │         如果是管道，传递给下一步     │
    │         {{prev_output}} = 上步输出  │
    └─────────────────────────────────────┘
         ↓
    更新 ProcessingResult (status: done)
```

#### 2.2.4 处理结果展示

**结果视图**：
- 默认显示处理结果（替换原文）
- Tab 切换：「处理结果」/「原文」
- 处理元信息：
  - 处理时间
  - 使用的模板/管道
  - Token 消耗（可选显示）

```
┌────────────────────────────────────────────────────┐
│  [处理结果]  [原文]                                 │
├────────────────────────────────────────────────────┤
│                                                    │
│  这是 AI 处理后的内容...                           │
│  （摘要/翻译/等）                                  │
│                                                    │
│  ──────────────────────────────────               │
│  处理时间: 2024-01-15 10:30                        │
│  使用管道: 阅读优化流程                            │
│  Token 消耗: 1,234                                 │
└────────────────────────────────────────────────────┘
```

**处理历史**：
- 一篇文章可以被多次处理（使用不同模板/管道）
- 保留历史处理记录
- 用户可以选择查看哪次处理的结果

#### 2.2.5 Feed 关联管道

**关联设置**：
- 在 Feed 详情/设置中可关联默认管道
- 设置后，该 Feed 下的文章可"一键处理"

**处理入口**：
- Feed 列表页："处理新文章"按钮
- 处理该 Feed 下未处理过的文章
- 使用关联的管道

## 3. 数据模型设计

### 3.1 新增数据表

#### 3.1.1 工艺模板表 (craft_templates)

```typescript
craftTemplates = sqliteTable('craft_templates', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  userId: integer('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
  name: text('name').notNull(),                    // 模板名称
  description: text('description'),                // 模板描述

  aiConfigId: integer('ai_config_id').notNull()    // 关联的 AI 配置
    .references(() => aiConfigs.id, { onDelete: 'restrict' }), // 禁止删除有关联模板的 AI 配置
  promptTemplate: text('prompt_template').notNull(), // Prompt 模板

  category: text('category').notNull().default('custom'), // 分类
  // 可选值: 'summarize' | 'translate' | 'filter' | 'analyze' | 'rewrite' | 'custom'

  createdAt: integer('created_at').notNull(),
  updatedAt: integer('updated_at').notNull(),
}, (table) => ({
  userIdIdx: index('craft_templates_user_id_idx').on(table.userId),
  categoryIdx: index('craft_templates_category_idx').on(table.category),
}))
```

#### 3.1.2 管道表 (pipelines)

```typescript
pipelines = sqliteTable('pipelines', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  userId: integer('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
  name: text('name').notNull(),                    // 管道名称
  description: text('description'),                // 管道描述

  // 管道步骤 (JSON 数组)
  steps: text('steps').notNull(),
  // 格式示例:
  // [
  //   { templateId: 1, order: 0, name: "摘要" },
  //   { templateId: 2, order: 1, name: "翻译" }
  // ]

  createdAt: integer('created_at').notNull(),
  updatedAt: integer('updated_at').notNull(),
}, (table) => ({
  userIdIdx: index('pipelines_user_id_idx').on(table.userId),
}))
```

#### 3.1.3 处理结果表 (processing_results)

```typescript
processingResults = sqliteTable('processing_results', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  userId: integer('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
  feedItemId: integer('feed_item_id').notNull().references(() => feedItems.id, { onDelete: 'cascade' }),

  // 使用的处理方式（二选一）
  pipelineId: integer('pipeline_id').references(() => pipelines.id, { onDelete: 'set null' }),
  templateId: integer('template_id').references(() => craftTemplates.id, { onDelete: 'set null' }),

  // 处理结果
  output: text('output'),                          // 最终输出
  stepsOutput: text('steps_output'),               // JSON: 每步输出记录
  // 格式示例:
  // [
  //   { step: 0, templateId: 1, output: "...", tokensUsed: 100 },
  //   { step: 1, templateId: 2, output: "...", tokensUsed: 150 }
  // ]

  // 状态
  status: text('status').notNull().default('pending'), // 'pending' | 'processing' | 'done' | 'error'
  errorMessage: text('error_message'),

  // Token 使用记录
  tokensUsed: integer('tokens_used'),

  createdAt: integer('created_at').notNull(),
  completedAt: integer('completed_at'),
}, (table) => ({
  userIdIdx: index('processing_results_user_id_idx').on(table.userId),
  feedItemIdIdx: index('processing_results_feed_item_id_idx').on(table.feedItemId),
  statusIdx: index('processing_results_status_idx').on(table.status),
}))
```

## 4. API 设计

### 4.1 工艺模板 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/craft-templates` | 获取模板列表 |
| GET | `/api/craft-templates/[id]` | 获取单个模板 |
| POST | `/api/craft-templates` | 创建模板 |
| PUT | `/api/craft-templates/[id]` | 更新模板 |
| DELETE | `/api/craft-templates/[id]` | 删除模板 |
| POST | `/api/craft-templates/[id]/test` | 测试模板 |

### 4.2 管道 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/pipelines` | 获取管道列表 |
| GET | `/api/pipelines/[id]` | 获取单个管道 |
| POST | `/api/pipelines` | 创建管道 |
| PUT | `/api/pipelines/[id]` | 更新管道 |
| DELETE | `/api/pipelines/[id]` | 删除管道 |
| POST | `/api/pipelines/[id]/test` | 测试管道 |

### 4.3 处理 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/process` | 处理文章（单篇/批量） |
| GET | `/api/processing-results` | 获取处理结果列表 |
| GET | `/api/processing-results/[id]` | 获取单个处理结果 |
| GET | `/api/feed-items/[id]/processing-results` | 获取文章的处理历史 |

**POST /api/process 请求体**：

```typescript
// 单篇处理
{
  feedItemId: number,
  templateId?: number,    // 使用单个模板
  pipelineId?: number,    // 或使用管道
}

// 批量处理
{
  feedItemIds: number[],
  templateId?: number,
  pipelineId?: number,
}
```

### 4.4 Feed 关联 API

| 方法 | 路径 | 说明 |
|------|------|------|
| PUT | `/api/feeds/[id]/pipeline` | 设置 Feed 关联管道 |
| DELETE | `/api/feeds/[id]/pipeline` | 移除关联 |
| POST | `/api/feeds/[id]/process-new` | 处理该 Feed 的新文章 |

## 7. 非功能需求

### 7.1 性能要求

- 单篇处理响应时间 < 30s（取决于 AI API）
- 批量处理并发数限制为 3，避免 API 过载
- 处理结果缓存，避免重复处理

### 7.2 安全要求

- 处理结果与用户关联，确保数据隔离
- AI API 调用使用用户配置的 API Key
- 防止 Prompt 注入攻击

### 7.3 用户体验

- 处理进度实时反馈
- 错误信息友好展示
- 支持重试失败的处理

## 8. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| AI API 调用失败 | 处理中断 | 整体失败 + 错误记录，用户可重试 |
| Token 消耗过大 | 成本增加 | 设置限制 + 提醒用户 |
| 处理时间过长 | 用户体验差 | 异步处理 + 进度展示 |
| Prompt 效果不佳 | 输出质量差 | 提供预设模板库 + 测试功能 (P1) |
| AI Config 被误删 | 关联模板失效 | API 预检查，禁止删除有关联模板的配置 |
| 处理历史数据增长 | 存储压力 | 暂全量存储，后续添加清理策略 |
