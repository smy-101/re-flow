## ADDED Requirements

### Requirement: 统一基础控件体系
系统SHALL为按钮、输入框、文本域、选择器、卡片和徽标提供统一的组件 API 与视觉状态。

#### Scenario: 不同页面复用基础控件
- **GIVEN** 认证页、设置页和列表筛选区同时使用基础控件
- **WHEN** 这些页面渲染相同语义的控件
- **THEN** 系统提供一致的尺寸、圆角、留白和状态反馈
- **AND** 页面无需重复定义同类控件的核心视觉规则

#### Scenario: 基础控件覆盖常见状态
- **GIVEN** 用户与基础控件交互
- **WHEN** 控件进入默认、悬停、聚焦、禁用、加载或错误状态
- **THEN** 系统提供一致的交互反馈
- **AND** 这些状态在亮色与暗色模式下都可辨识

### Requirement: 统一交互容器体系
系统SHALL为弹窗、抽屉、下拉菜单和危险操作确认提供统一的交互容器行为。

#### Scenario: 弹出层交互一致
- **GIVEN** 用户打开弹窗、抽屉或下拉菜单
- **WHEN** 用户查看标题、内容区和关闭入口
- **THEN** 系统使用一致的结构层级与关闭行为
- **AND** 弹出层表面样式与当前主题保持一致

#### Scenario: 危险操作需要确认
- **GIVEN** 用户触发删除或批量确认类操作
- **WHEN** 系统展示确认容器
- **THEN** 系统突出危险级别并提供明确的确认与取消动作
- **AND** 不再依赖页面各自实现的非统一确认样式

### Requirement: 统一反馈组件体系
系统SHALL为行内提示、Toast、骨架屏和状态提示提供统一的反馈组件。

#### Scenario: 操作结果反馈一致
- **GIVEN** 用户执行成功、失败或信息提示类操作
- **WHEN** 系统展示反馈信息
- **THEN** 系统使用统一的反馈组件和状态语义
- **AND** 反馈颜色、图标和文案层级在不同页面保持一致

#### Scenario: 加载与占位一致
- **GIVEN** 页面需要等待数据或异步操作完成
- **WHEN** 系统展示加载或占位内容
- **THEN** 系统使用统一的骨架屏或加载样式
- **AND** 不再由各页面单独拼装互不一致的加载反馈
