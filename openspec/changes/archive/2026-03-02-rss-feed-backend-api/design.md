## Context

The re-flow RSS reader currently has a fully functional UI (`app/(dashboard)/feeds/page.tsx`, `components/feeds/`, `components/items/`) backed by in-memory mock data from `lib/mock-data.ts`. The database only contains a `users` table with existing authentication infrastructure (JWT + bcrypt).

The mock data includes:
- `Feed` entity with id, userId, title, feedUrl, siteUrl, description, category, timestamps, unreadCount
- `FeedItem` entity with id, feedId, title, link, content, publishedAt, isRead, isFavorite, author, readingTime
- Mock API functions simulating CRUD operations with network delay

This change implements real persistence and API endpoints to replace the mock data layer.

## Goals / Non-Goals

**Goals:**
- Persist feeds and feed items in SQLite database via Drizzle ORM
- Create RESTful API routes using Next.js App Router conventions
- Maintain API compatibility with existing component interfaces (drop-in replacement)
- Secure all endpoints with existing JWT authentication
- Support RSS feed URL validation before subscription

**Non-Goals:**
- RSS feed fetching/scheduling (will be added later as background job)
- Feed update notifications/real-time updates
- Advanced filtering/search beyond existing UI requirements
- OPML import/export

## Decisions

### Database Schema Design

**Choice**: Extended `lib/db/schema.ts` with two new tables mirroring mock data types

```typescript
feeds (id, userId, title, feedUrl, siteUrl, description, category, createdAt, lastUpdatedAt)
feed_items (id, feedId, userId, title, link, content, publishedAt, isRead, isFavorite, author, readingTime, createdAt)
```

**Rationale**:
- Directly maps to existing `Feed`/`FeedItem` interfaces in `lib/mock-data.ts`
- Minimal migration from mock to real data
- `userId` on `feed_items` enables efficient user-scoped queries without joins

**Alternatives considered**:
- Items without userId: Would require JOIN with feeds for every query → rejected for performance
- Separate categories table: Overkill for simple string tags → rejected for simplicity

### API Route Structure

**Choice**: RESTful endpoints under `app/api/` following Next.js 16 App Router conventions

```
GET/POST    /api/feeds
GET/PUT/DEL /api/feeds/[id]
GET         /api/feeds/[id]/items
POST        /api/feeds/validate
GET         /api/categories
GET         /api/items (with query filters: feedId, isRead, isFavorite)
GET         /api/items/[id]
PATCH       /api/items/[id]/read
POST        /api/items/[id]/favorite
```

**Rationale**:
- Aligns with existing mock API function signatures
- Standard REST patterns for CRUD
- Separates operations (read toggle vs favorite toggle) for clarity

**Alternatives considered**:
- GraphQL: Overkill for this scope, adds complexity
- tRPC: Would require significant refactoring of existing components

### RSS Parser Library

**Choice**: `rss-parser` (npm package)

**Rationale**:
- Lightweight, well-maintained, supports RSS/Atom
- Browser and Node.js compatible
- Handles malformed feeds gracefully

**Alternatives considered**:
- `parser`: Heavier, more features than needed
- Custom parser: High maintenance burden

### Authentication Integration

**Choice**: Reuse existing `lib/auth/` middleware with JWT verification

**Rationale**:
- Consistent with established patterns
- JWT already stored in HTTP-only cookies
- Middleware extracts `userId` for database queries

### API Client Layer

**Choice**: Create `lib/api/` directory with functions mirroring `lib/mock-data.ts`

**Rationale**:
- Drop-in replacement for existing imports
- Preserves component logic (only import path changes)
- Single source of truth for API communication

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| RSS parsing errors for malformed feeds | Wrap in try/catch, return validation errors to user |
| Database performance with many items per feed | Add indexes on `feedId`, `userId`, `publishedAt`; pagination on list endpoints |
| Feed URLs becoming invalid | Validation endpoint checks before subscription; graceful degradation |
| Migration breaking existing data | N/A - new tables, no existing feed data to migrate |
| Type drift between mock and real APIs | Use shared types from Drizzle schema inference |
