## Context

当前应用使用顶部导航栏布局，包含 Logo、搜索框、导航链接（我的订阅、我的阅读）和用户菜单。随着收藏、AI 助手、中转等功能的规划，顶部导航空间将变得拥挤，且难以有效展示更多入口。现代仪表盘应用普遍采用左侧边栏布局，这种方式在可扩展性和用户体验方面更具优势。

## Goals / Non-Goals

**Goals:**
- 将主导航从顶部导航栏迁移至左侧可折叠边栏
- 提供桌面端和移动端两种不同的导航体验
- 保持现有功能的完整性和用户体验连贯性
- 为未来功能扩展预留空间

**Non-Goals:**
- 不修改现有的 API 端点
- 不改变数据库结构
- 不重新设计页面内容区域
- 不添加权限控制逻辑

## Decisions

### 1. 边栏状态管理

**决策：** 使用 LocalStorage 持久化边栏展开/收起状态

**理由：**
- 用户体验更佳，无需每次访问重新设置
- 实现简单，无需后端支持
- 符合现代 Web 应用惯例

**替代方案：**
- URL 参数：可分享但不符合此场景
- Session only：每次重置，体验不佳

### 2. 图标库选择

**决策：** 使用 lucide-react 图标库

**理由：**
- 树摇优化，按需导入，打包体积小
- 图标风格统一，符合现代设计规范
- TypeScript 支持完善
- 社区活跃，维护良好

### 3. 侧边栏布局组件设计

**决策：** 分离 `Sidebar` 和 `SidebarItem` 组件

**理由：**
- `Sidebar` 负责整体布局、状态管理、折叠动画
- `SidebarItem` 负责单个导航项的渲染和交互
- 符合单一职责原则
- 便于未来扩展和测试

### 4. 移动端导航模式

**决策：** 使用抽屉式导航，而非堆叠导航

**理由：**
- 移动端屏幕空间宝贵，抽屉可以节省内容区域
- 交互模式与主流应用一致，学习成本低
- 可以优雅地处理更多导航项

### 5. 导航路由映射

**决策：** 使用静态配置而非动态渲染

**理由：**
- 导航项数量有限且相对固定
- 类型安全，编译时即可检查路由
- 便于国际化或未来扩展

### 6. 激活状态显示

**决策：** 组合使用左侧边框、背景色和图标颜色变化

**理由：**
- 视觉层次清晰，多种视觉线索互补
- 折叠状态下仍能通过边框和图标颜色识别
- 符合用户心智模型

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| 边栏收起后，用户可能找不到功能 | 添加清晰的提示，图标悬停显示完整文字 |
| 移动端抽屉与现有移动菜单冲突 | 统一移动端导航逻辑，使用单一控制点 |
| LocalStorage 可能在隐私模式下不可用 | 降级处理，使用默认展开状态 |
| 过渡动画可能影响性能 | 使用 CSS transform 优化，避免重排 |

### 权衡

- **扩展性 vs 简洁性**：选择支持未来功能扩展的设计，增加了初始复杂度
- **持久化 vs 隐私**：LocalStorage 提升用户体验，但需处理隐私模式场景
- **动画流畅性 vs 性能**：边栏动画提升体验，但需确保使用 CSS transform 避免性能问题
