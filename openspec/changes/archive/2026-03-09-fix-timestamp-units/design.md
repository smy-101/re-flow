## Context

当前 AI 配置、工艺模板、管道三组数据在 SQLite schema 中按 Unix 秒级时间戳建模，但相关 API 路由在创建、更新、测试状态变更时写入 `Date.now()` 毫秒值，前端组件又按秒级值乘以 1000 渲染，导致设置页出现五万多年后的错误日期。该问题横跨 `app/api/ai-configs`、`app/api/craft-templates`、`app/api/pipelines` 与 `components/ai`、`components/craft`、`components/pipeline`。相关展示组件均为 Client Component，继续消费现有 API，不新增 app/ 路由。

文件变更清单预计包括：相关 API 路由、共享时间工具、显示时间的 Client 组件、数据库迁移或修复脚本，以及对应的 API/组件测试。

## Goals / Non-Goals

**Goals:**
- 为 AI 配置、工艺模板、管道建立单一时间语义：持久化与接口返回统一为秒级时间戳。
- 停止新写入的毫秒级脏数据，并修复已落库的错误数据。
- 让设置页相关时间展示在迁移前后都保持正确且可测试。

**Non-Goals:**
- 不调整与本问题无关的 UI 样式或文案。
- 不扩散修改到未确认受影响的业务域，除非需要复用统一时间工具。

## Decisions

### 决策一：以后端持久化语义为准，统一使用秒级时间戳

数据库 schema 已以 `strftime('%s', 'now')` 定义秒级默认值，RSS 与处理队列等现有模块也主要按秒级处理。沿用秒级可避免扩大影响面，并与当前表结构保持一致。

备选方案：改为全站毫秒级。
放弃原因：需要系统性修改更多 schema、显示逻辑与测试，收益低于成本。

### 决策二：在写入层集中收敛时间生成

相关路由改为统一调用共享秒级时间函数，而不是各处直接写 `Date.now()`。这样可以减少再次混入毫秒值的风险，并让测试断言更直接。

备选方案：仅在前端格式化时兼容秒/毫秒。
放弃原因：只能掩盖问题，无法阻止脏数据继续进入数据库。

### 决策三：增加一次性数据修复，把已写入的毫秒值归一为秒级

对 `ai_configs`、`craft_templates`、`pipelines` 的 `created_at`、`updated_at` 以及 `ai_configs.last_error_at` 执行条件修复，按数值范围识别毫秒值并除以 1000 取整。这样历史数据与新数据能在同一语义下工作。

备选方案：不修库，只在前端做兼容。
放弃原因：历史数据仍不可信，且会污染后续排序、比较与调试。

### 决策四：前端保留短期容错格式化

展示层通过共享格式化入口兼容秒级与遗留毫秒级值，保证迁移窗口内页面稳定；待全部数据修复完成后，该容错仍可保留为防御性逻辑。

## Risks / Trade-offs

- [迁移遗漏字段] → 在 design 与任务中显式列出所有受影响字段，并通过查询验证修复结果。
- [旧组件仍直接 `new Date(timestamp)`] → 使用搜索与测试覆盖时间格式化调用点，优先替换设置页相关组件。
- [秒/毫秒混用测试继续存在] → 为创建、更新、测试状态更新分别增加断言，要求返回值与持久化值均为秒级整数。

## Migration Plan

1. 先修改 API 写入逻辑，阻止新增毫秒值。
2. 再执行数据库修复，将三张表中的毫秒级历史值转换为秒级。
3. 更新前端格式化入口并验证设置页展示。
4. 通过测试和抽样 SQL 查询确认新旧数据均按秒级工作。

回滚策略：若迁移异常，可先回滚应用代码并恢复修复前数据库备份；若仅前端兼容逻辑异常，可单独回退显示层改动而不影响已修正的数据语义。

## Open Questions

- 是否同时把 items 详情页等其他疑似直接按毫秒渲染秒级值的页面纳入本次修复，还是作为后续独立变更处理。
